# NodeLoc Daily Maintainer Plan

This document records the high-level direction only. It is intentionally not a detailed implementation spec.

## Goal

Build a daily maintainer for the 6 NodeLoc accounts. It should keep routine account activity healthy without posting spam or generating forum content.

## Core Approach

Use a hybrid model:

- Protocol requests for lightweight, deterministic actions.
- Playwright browser automation for real reading behavior that Discourse normally records through the frontend.

## Daily Flow

1. Load account cookies from `accounts.json`.
2. Check whether each cookie is still valid.
3. Run daily check-in.
4. Query current user stats and trust level.
5. Decide which accounts need reading time or topic/post progress.
6. For accounts that need reading, launch Playwright with the account cookie.
7. Open real topics, stay on pages, scroll slowly, and move between topics.
8. Query stats again after reading.
9. Write a concise daily report.

## Protocol Layer

Use plain HTTP requests for:

- Check-in.
- Cookie validity checks.
- Current user lookup.
- Trust level lookup.
- User summary stats.
- Topic list discovery.
- Final report data.

## Playwright Layer

Use real browser automation for:

- Cookie-based login.
- Opening topic pages.
- Slow scrolling.
- Staying on readable content for a natural amount of time.
- Letting Discourse record `topics_entered`, `posts_read_count`, and `time_read` through its normal frontend timing flow.

## Boundaries

Allowed:

- Daily check-in.
- Reading assistance.
- Stats collection.
- Progress reports.
- Low-frequency, non-coordinated likes may be considered later.

Not allowed:

- Automatic replies.
- Automatic posting.
- Coordinated likes across accounts.
- Multiple accounts boosting the same post or user.
- Fake large reading-time submissions.
- High-frequency activity that looks like spam.

## Notes From Source Research

Discourse records reading progress through frontend timing reports to `/topics/timings`.

Important metrics:

- `days_visited`: increases when a daily visit record is created.
- `topics_entered`: increases when a user first views a topic.
- `posts_read_count`: increases when post timing is first recorded for posts.
- `time_read`: increases from real elapsed time between timing updates, with server-side limits.

This means reading progress should be generated by real browser behavior, not by manually submitting large artificial timing values.

## Current Status

- Daily check-in script exists.
- 6 account cookies are configured.
- Cookie-based browser login has been verified for all 6 accounts.
- Same-day local check-in state is stored in `.nodeloc_state.json`.
- The daily maintainer itself has not been implemented yet.
